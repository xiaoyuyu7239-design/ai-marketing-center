# 绘卖邀请内测上线运行手册

更新时间：2026-07-16

适用范围：首波 2 家、稳定后逐步扩至 5–10 家受邀商户的单机单实例 Next.js + SQLite 部署。本手册是邀请内测的值守基线，不代表已经具备多副本或公开付费上线能力。

## 1. 内测运行边界

- Web 服务只运行一个实例；不得启用 PM2 cluster、多个容器副本或让多台主机同时写同一个 SQLite 数据目录。
- `sqlite.db`、`uploads/`、`output/` 必须位于同一个持久化数据根目录，但备份必须写到另一个挂载点，最好再同步到异机或对象存储。
- 新版本未通过 readiness、备份恢复演练和人工冒烟前，不接入邀请用户流量。
- 生产密钥只通过部署 secret 注入。数据备份不包含 `.env`、证书和部署 secret，它们必须由独立的密钥管理流程保管。
- SQLite 中可能包含模型配置和访问凭据，因此整套备份按高敏感数据处理：最小权限、传输加密、静态加密、禁止公开链接。

内测阶段目标：在“每 6 小时备份且每份都完成异机同步”连续验证后，整机灾难 RPO（最多可接受的数据丢失）不超过 6 小时；完整 RTO（从故障发现到验收并切流）不超过 2 小时。调度或异机同步尚未验收时，不得宣称已达到该目标。

## 2. 上线前环境基线

必须明确配置以下路径和阈值：

```bash
APP_DATA_DIR=/data
APP_MIGRATIONS_DIR=/app/drizzle
BACKUP_DIR=/backups
TZ=Asia/Shanghai
# 构建时注入的完整 40/64 位 release commit SHA，同时写入 Next BUILD_ID
HUIMAI_CODE_VERSION=replace-with-build-commit-sha
HUIMAI_PREFLIGHT_READY_URL=http://127.0.0.1:3000/api/health/ready
HUIMAI_BACKUP_EVIDENCE_FILE=/release-evidence/backup-verification.json
HUIMAI_BACKUP_EVIDENCE_SHA256=replace-with-file-sha256
HUIMAI_BACKUP_OFFSITE_RECEIPT_FILE=/release-evidence/offsite-backup-receipt.json
HUIMAI_BACKUP_OFFSITE_RECEIPT_SHA256=replace-with-file-sha256
CLIPFORGE_MIN_FREE_DISK_BYTES=2147483648
# 每商户累计上传上限，单位 bytes；不配置时默认 2 GiB
HUIMAI_MERCHANT_UPLOAD_MAX_BYTES=2147483648
```

生产环境还必须配置真实运营主体、联系渠道、数据保留期限、模型服务商披露，以及写入成片隐式标识的服务提供者名称或编码：

```bash
HUIMAI_LEGAL_ENTITY=实际运营主体
HUIMAI_LEGAL_CONTACT=真实客服或隐私联系渠道
HUIMAI_DATA_RETENTION_DAYS=30
HUIMAI_AI_PROVIDER_DISCLOSURE=实际启用的模型服务商及必要说明
HUIMAI_AIGC_SERVICE_PROVIDER=服务提供者名称或编码
```

`HUIMAI_DATA_RETENTION_DAYS` 目前用于启动门禁和隐私披露，不会自动删除服务日志、项目或备份。首次邀请用户前必须在部署平台落实同期限的日志保留策略，并确定账号/项目删除、第三方数据处理、备份轮换删除和处理留证流程；做不到时应先调整真实承诺，不能只改环境变量。

邀请内测必须显式关闭公开注册、使用邮箱白名单，并为后台配置两份相互独立的强凭据：

```bash
HUIMAI_DEPLOYMENT_MODE=saas
HUIMAI_PUBLIC_BASE_URL=https://实际生产域名
HUIMAI_INVITE_ONLY=1
HUIMAI_PUBLIC_SIGNUP=0
HUIMAI_INVITE_EMAILS=owner-a@example.com,owner-b@example.com
CLIPFORGE_ADMIN_PASSWORD=至少16位强随机值
CLIPFORGE_ADMIN_SESSION_SECRET=至少32位、且不同于后台密码的强随机值
```

不要在生产配置 `HUIMAI_INVITE_CODES`、`CLIPFORGE_INVITE_CODES` 或 `CLIPFORGE_SINGLE_USER=1`。模型密钥使用控制面约定的部署 secret：LLM、图片、视频、TTS 的 primary/fallback 分别使用各自专属变量，例如 `CLIPFORGE_LLM_API_KEY` 与 `CLIPFORGE_LLM_FALLBACK_API_KEY`。生产不会让 fallback 回退复用 primary 或通用旧密钥；后台数据库和浏览器只保存预期的 `secretRef`，不保存真实密钥，也不能在控制面改成另一个已允许但角色不匹配的引用。自定义模型 API 主机还必须加入 `HUIMAI_MODEL_ENDPOINT_HOSTS`，多个主机用逗号分隔。

豆包语音 TTS 使用语音新版控制台签发的独立 API Key，不得复用火山方舟 ARK Key。主端点配置为 `CLIPFORGE_TTS_PROVIDER=volcengine`、`CLIPFORGE_TTS_BASE_URL=https://openspeech.bytedance.com/api/v3/tts`、`CLIPFORGE_TTS_MODEL=seed-tts-2.0`，`CLIPFORGE_TTS_VOICE` 必须填写当前账号已开通的真实音色 ID。

模型配置里的名称、版本字符串或 deployment 名称本身不构成“固定版本”证据。每个将上线的 primary/fallback 都必须填写 `deploymentRevision`、`revisionEvidenceFile` 与 `revisionEvidenceSha256`；把脱敏后的供应商控制台/API 回执放入只读挂载的 `HUIMAI_MODEL_REVISION_EVIDENCE_DIR`。证据 JSON 必须采用 `config/model-revision-evidence.schema.example.json` 的精确结构，并绑定 `endpointHost`、`model`、`deploymentRevision`、`faultDomain`、`immutable`、`evidenceType`、`sourceReference`、`issuedAt`。生产预检会逐文件核对哈希和字段并拒绝已知浮动别名；证据内容仍须由负责人确认确实证明上游不可变/锁定映射。`latest`、展示别名和可被重新指向的 deployment 均不合格。

主备“跨供应商”按实际 endpoint hostname 的故障域判断，不采信可任意填写的 provider 文本。常见供应商使用内置 hostname 映射；自定义主机必须通过 `HUIMAI_MODEL_ENDPOINT_FAULT_DOMAINS=host=domain,...` 声明真实故障域。同一上游的不同域名必须映射到同一个值；名称不同但实际落在同一故障域的主备会失败关闭。

说明：

- `/data` 必须是可写持久化卷，包含 `sqlite.db`、`uploads/`、`output/`。
- `/app/drizzle` 必须随当前版本发布；迁移目录缺失或迁移执行失败时，服务会失败关闭，不能带着半套 schema 接收请求。
- 当前邀请内测迁移链必须包含 27 条 SQL，并覆盖 `0018_persistent_jobs.sql` 至 `0026_motion_job_schema_upgrade.sql`；预检会同时核对持久任务、工作流计费、子项租约、Golden 媒体任务、TTS one-shot、分镜动态任务/资格评估与旧 `output_asset_id` 草案向 `video_clips` 的 0026 升级。
- `/backups` 必须是独立挂载，不能位于 `/data` 内，也不能让 `/data` 位于它内部；Web 容器长期以只读方式挂载，仅一次性专用维护任务在已冻结写入的窗口内短时读写挂载。
- `/model-revision-evidence` 与 `/release-evidence` 必须从宿主机只读挂载；前者保存脱敏的模型不可变 revision/deployment 证据，后者保存真实 HTTPS 边缘请求体与备份/异机回执证据。证据应由专用维护流程生成、审阅并计算 SHA-256，Web 容器不得改写。
- readiness 默认要求至少 2 GiB 可用空间；只有经过容量评估后才调整 `CLIPFORGE_MIN_FREE_DISK_BYTES`。
- `HUIMAI_MERCHANT_UPLOAD_MAX_BYTES` 是单个商户所有可归属项目素材与商品图的累计上限，默认 2 GiB；它不是单文件上限，也不替代数据盘磁盘告警。
- `FFMPEG_PATH`、`FFPROBE_PATH` 可显式指定；不指定时使用系统可执行文件。FFmpeg 必须支持 `drawtext`。

### 反向代理原始请求体、来源 IP 与临时盘门禁（首邀前必须）

应用路由对文件数、单文件、解析后总量和商户累计空间的校验，发生在 HTTP body 已进入应用或 `formData` 解析之后，不能独立抵御 chunked/无 `Content-Length` 的超大请求。首次邀请用户前，HTTPS 网关或反向代理必须在数据进入 Next.js 之前以流式方式执行原始请求体硬上限；限流和商户额度都不能替代这一层。

- 仓库提供两份必须配套安装的基线：`config/nginx-invite-beta.conf.example` 放在 Nginx `http{}`，`config/nginx-huimai-proxy.inc.example` 安装为 `/etc/nginx/snippets/huimai-proxy.inc`。默认请求体仅 2 MiB；`/api/upload` 与 `/api/products/upload` 为 81 MiB，项目 BGM 为 21 MiB，只有项目 materials 路由放宽至 161 MiB。上传 location 在接收 body 前执行每 IP 连接数和 20 次/分钟限流，避免匿名请求占满临时盘。
- `proxy_request_buffering on` 使 Nginx 完整接收并执行硬上限后才转发应用。`client_body_temp_path` 必须位于独立受配额与监控的文件系统；邀请内测基线建议至少 4 GiB 硬配额、80% 告警。若平台无法提供等价约束，应先降低上传上限或改为分片/对象存储直传。
- 应用端口只绑定 `127.0.0.1` 且防火墙不允许公网直连。生产必须设置 `HUIMAI_TRUST_PROXY=1`、`HUIMAI_CLIENT_IP_HEADER=x-real-ip`；Nginx snippet 用 `$remote_addr` 覆盖 `X-Real-IP`/`X-Forwarded-For` 并清空 `CF-Connecting-IP`、`True-Client-IP`、`Forwarded`。若前面还有 CDN/LB，必须先按其官方出口 IP 配置 Nginx `real_ip`，绝不能直接信任客户端传来的来源头。
- 在部署机运行 `nginx -T`，对完整展开输出计算 SHA-256 并写入 `HUIMAI_EDGE_GATEWAY_CONFIG_SHA256`。同时配置真实 HTTPS origin、`HUIMAI_EDGE_MAX_BODY_BYTES=168820736`、材料路由探针路径 `/api/project/<安全ID>/materials` 和证据输出，再运行 `pnpm ops:verify-edge`。
- 探针先发送 `max-1` 字节的 `Expect: 100-continue` 正向对照，该请求可以因未鉴权/格式错误返回 4xx，但不能是 0 或 413；随后发送 `max+1` 的固定 `Content-Length` 与 chunked 请求，两者都必须由边缘返回 413。证据 JSON 精确绑定 origin、探针路径、Nginx 展开配置 hash、上限、三次状态和验证时间。
- 将输出 SHA-256 写入 `HUIMAI_EDGE_BODY_LIMIT_EVIDENCE_SHA256` 后再运行 `ops:preflight`；预检会校验上述精确字段、文件哈希及七天新鲜度。探针只证明原始 body 门禁，合法业务 multipart 仍须用内部商户账号单独冒烟。任一项未通过，继续 No-Go。

### 上线预检

先用当前版本在隔离端口启动一次，使数据库完成迁移和旧模型配置脱敏规范化；随后执行：

```bash
HUIMAI_PREFLIGHT_READY_URL=http://127.0.0.1:3100/api/health/ready pnpm ops:preflight
```

预检会以只读方式检查 SQLite，不会显示任何密钥。它覆盖邀请制、后台凭据、法律/AIGC 配置、数据与备份目录隔离、磁盘、至 `0026` 的 27 条迁移链及分镜动态最终 schema、FFmpeg/FFprobe、角色专属 secretRef、HTTPS/主机白名单、按 endpoint 映射的主备故障域、结构化模型 revision 证据、带正向对照和网关配置 hash 的边缘 body 证据、正式备份/冻结窗口/异机版本回执，以及当前代码/Prompt/Golden Set 对应的发布证据。它能证明“当前配置绑定了哪份证据”，不能替代负责人审阅供应商回执真实性或合法业务 multipart 冒烟。任何 `FAIL` 都阻止接入邀请用户；`HUIMAI_PREFLIGHT_READY_URL` 是必填门禁，且只能是当前容器 loopback 上精确的 `/api/health/ready`，不能用公网/CDN 或另一实例替代。

Web 容器启动时挂载可写 `/data`、独立**只读** `/backups`，以及只读 `/model-revision-evidence`、`/release-evidence`。例如：

```bash
docker run --name huimai \
  --env-file /host/huimai-production.env \
  -v /host/huimai-data:/data \
  -v /host/huimai-backups:/backups:ro \
  -v /host/huimai-model-revision-evidence:/model-revision-evidence:ro \
  -v /host/huimai-release-evidence:/release-evidence:ro \
  -p 127.0.0.1:3000:3000 \
  huimai@sha256:replace-with-verified-image-digest
```

生产部署必须使用远端 CI 推送并已验收的不可变镜像 digest，不使用浮动标签作为最终放行依据。备份脚本必须随同一 release 镜像或独立运维镜像提供；一次性维护任务显式把 `/data` 只读、`/backups` 读写挂载，Web 不得为了备份临时改成长期开写。不要把数据库复制到 Web 容器可写层。

按当前 Dockerfile/流水线构建的镜像应同时包含预检脚本和 `verify-edge-body-limit.mjs` 边缘探针，但仍须由远端 CI 和镜像冒烟确认。注入真实生产环境变量、挂载真实数据与备份卷并等待 readiness 后，在容器内执行最终门禁：

```bash
docker exec \
  -e HUIMAI_PREFLIGHT_READY_URL=http://127.0.0.1:3000/api/health/ready \
  huimai node /app/scripts/preflight-beta.mjs
```

`.github/workflows/docker-publish.yml` 只会在依赖锁定安装、lint、全量 TypeScript 检查和测试通过后构建镜像。构建前运行 `scripts/verify-release-source.mjs --require-git-tracked`，拒绝任何不属于声明 commit 的发布白名单文件；随后在一次性空数据容器中验证 27 条迁移、liveness/readiness、制品 BUILD_ID、邀请状态、`/start`、Next/public 静态资源、FFmpeg、备份/预检/边缘探针工具与抠图依赖。成功后才把同一已测 image ID 推送到完整 commit 标签，反查 GHCR manifest/config digest，并上传 `release-evidence.json`、源码清单和 raw manifest。公开仓库还必须生成 provenance attestation；用户所有的私有仓库当前不支持 GitHub Artifact Attestations，流水线会显式记录 `skipped-private-repository`，不得为取得该能力而把邀请内测仓库改为公开。生产只部署 `image@sha256:...`。CI 冒烟使用测试配置，不能替代上述真实生产数据预检和人工业务冒烟。

Docker context 以 `**` 默认拒绝，只按 `config/release-source-paths.txt` 和 Dockerfile 的显式 COPY 放行 Web 构建输入；`pnpm ops:verify-release-source` 会生成逐文件大小/权限/SHA-256 和确定性源码摘要，并拒绝数据、密钥、归档、大文件、特殊文件、大小写/Unicode 冲突、符号链接和 `COPY .` 漂移。

`pnpm build` 还会清理并校验 `.next/standalone`：发布包不得包含业务 `data/`、Git 历史、`docs/` 截图、版本归档、项目/提示词/产品文档、Codex 记忆、`.env*` 等指定敏感内容，所有符号链接必须存在且留在发布根内，默认总大小不得超过 512 MiB。Next.js 正常追踪的少量运行时源码不属于该“零源码”承诺。该检查用于防止真实商户 SQLite/媒体和私密研发归档误入镜像或桌面包，不得从发布流水线删除，也不得仅为通过构建而随意调高 `HUIMAI_MAX_STANDALONE_BYTES`。

## 3. 限流、上传空间与工作流额度

### 模型、付费与高 CPU 入口限流

所有有商户身份的模型生成、付费能力和高 CPU 入口都同时消耗“商户”和“来源 IP”两组桶，每组又同时检查 1 分钟突发窗口与 1 小时持续窗口。任意一个窗口超限都返回 HTTP 429，并携带 `Retry-After`、`X-RateLimit-Remaining` 和 `X-RateLimit-Reset`；客户端和值守人必须按 `Retry-After` 等待，不得立即换 IP、换商户或新建幂等键继续重放。

| 能力预置 | 商户：分钟 / 小时 | IP：分钟 / 小时 | 运行说明 |
|---|---:|---:|---|
| LLM 主生成 | 6 / 30 | 60 / 300 | 脚本、文案等主生成链路 |
| 辅助模型 / manifest | 10 / 60 | 100 / 600 | 较轻模型任务和批量任务创建 |
| 图片生成 | 12 / 40 | 120 / 400 | 最多 9 项的批量链路保留少量可控重试空间 |
| 视频生成 | 12 / 15 | 120 / 150 | 正常 9 镜批量加有限重试，第 13 次分钟请求将被拦截 |
| 成片合成 | 5 / 20 | 50 / 200 | 同时另受持久任务队列上限约束 |
| 高 CPU 处理 | 4 / 20 | 40 / 200 | 抠图等本地高开销链路 |

无商户身份的旧控制面鉴权入口只能按 IP 做双窗口：供应商连通性测试默认 10/60（分钟/小时），付费 TTS 试听默认 6/30。这类入口不得开放给未鉴权访客。当前限流桶保存在单进程内存中；增加实例前必须迁移到 Redis 等共享原子存储。

### 商户累计上传空间

- 上传路由在写盘前核算该商户已有素材，并预留当前请求的全部字节；并发请求不能同时穿透 2 GiB 默认上限。
- 超过上限返回 HTTP 413 `MERCHANT_UPLOAD_QUOTA_EXCEEDED`；无法安全核算目录时失败关闭，返回 HTTP 503 `MERCHANT_UPLOAD_ACCOUNTING_FAILED` 和 `Retry-After: 5`。不要把 503 当成上传成功。
- 核算范围是数据库中归属于该商户的项目素材与商品图目录。无法归属的旧版孤儿目录不会被计入，应在内测前完成一次人工容量盘点。
- 预留锁也是单进程内状态，因此“单机单实例”是硬前提，不是部署建议。

### 工作流额度与幂等

- 月度生成额度来自商户套餐额度加运营赠送额度，按自然月核算。一次脚本 workflow、一个 1–9 项的图片/视频批量 manifest，或一个请求付费 TTS 的合成 workflow，都只能创建一条父额度流水；批量子项不得按镜数重复扣减。
- `operationId`/`Idempotency-Key`、项目、任务类型、manifest 和请求摘要会被冻结。相同内容重放只返回原状态；同一幂等键换内容返回冲突，不得换新键规避正在运行或提交结果不明的任务。
- 父任务的所有子项失败时释放该次额度；部分成功或全部成功计 1 次。合成仅在实际产出并持久化付费 TTS 音频后计 1 次；免费 TTS、纯 FFmpeg 或未产出付费音频的失败不计该次付费能力额度。
- 批量子项 pending 最长 30 分钟，running 租约为 120 秒、每 30 秒心跳续租；过期项由恢复器收口。开启多实例前必须把额度锁、租约和上传预留一起改为可跨进程协调的实现。

## 4. 持久任务与提交结果不明处置

### Agent 主备降级边界

放行行为必须按服务端明确错误分类执行，不能因“有 fallback”就尝试掩盖所有失败：

- 只有幂等任务中的显式 transient 错误（例如网络失败、超时、429、供应商 5xx）、可安全重做的 parse/空响应错误，或已知可接受且能证明请求未被受理的 billing/quota 错误，才允许转入已评测通过的 fallback。
- safety/policy 拦截、config/auth/模型不存在等配置错误、其余 client 4xx、unknown/未分类错误和 `submission_uncertain` 一律 fail-close；不得改用本地模板、默认模型或硬编码候选绕过。
- 图片、视频、付费 TTS 等一次性付费提交，只要出现网络失败、超时、5xx、损坏的 2xx 响应或任何“可能已受理”状态，就进入 `submission_uncertain`，不得跨供应商重提。只有供应商响应能明确证明未受理时，才可按预先批准的错误分类决定是否降级。

首次邀请用户前必须逐类演练并用 requestId、attempt、错误分类、fallback 原因、供应商任务/账单和最终状态交叉留证。任何 safety/config/client 4xx/unknown 被降级，或一次性付费提交被重复发起，均为 No-Go。

### 合成任务

- 合成 `POST` 只校验并冻结输入、事务入队，然后返回 HTTP 202。前端要保留同一 `Idempotency-Key` 和 `compositionId`，按响应中的 `Location`/`Retry-After` 轮询，直到终态；页面刷新、网络中断或 202 响应丢失都不是创建新键的理由。
- 单商户最多 2 个未完成任务，全局最多 20 个；超限返回 HTTP 429 和 `Retry-After: 10`。当前单实例 worker 串行执行。
- running 租约为 90 秒、每 30 秒心跳。进程崩溃后最多做一次租约恢复；旧 worker 的租约 token 无权覆盖新 worker 结果。只能取消 pending，running 的 FFmpeg 当前不强制杀死，取消返回 409。

### 付费 TTS 与 `submission_uncertain`

- 付费 TTS 的提交类 POST 不做自动网络重试。断网、超时、供应商 5xx，或 2xx 响应损坏/无法证明是否受理时，统一视为“可能已受理”；不得原样重提或切换另一付费供应商，先根据供应商任务 ID/账单对账。可确定未受理的 4xx 才按普通失败处置。
- 在普通合成中，付费 TTS 提交结果不明时不重试；只允许显式配置的免费本地/Edge TTS 作非计费降级，否则跳过该镜配音。值守人仍需核对原付费供应商账单，因为“未收到音频”不等于“供应商未计费”。
- Golden 媒体/TTS 评测任务会先持久化为 `submitting`，TTS 只调用一次 one-shot。无法确认结果时将终止为 `submission_uncertain`，不会回到 pending 重提；只有已经原子落盘且通过深度校验的音频与评测记录，才能在恢复时收敛为 `succeeded`。其他 `submission_uncertain` 必须人工对账，且在解决前阻断模型晋级。

## 5. 健康检查

### Liveness

```text
GET /api/health/live
```

- 成功：HTTP 200，`{"status":"ok"}`。
- 只证明 Next.js 进程能响应，不访问数据库、数据目录或 FFmpeg。
- 用于容器进程存活检查；不要因为外部依赖短暂故障不断重启进程。

### Readiness

```text
GET /api/health/ready
```

- 成功：HTTP 200，`{"status":"ready"}`。
- 失败：HTTP 503，`{"status":"not_ready"}`。
- 服务端实际检查数据库查询、数据目录真实写入、磁盘余量、FFmpeg 与 FFprobe；公开响应不会暴露文件路径、阈值或二进制错误。
- 用于部署放行和负载均衡摘流。所有响应均带 `no-store`，不得由 CDN 缓存。
- 生产构建把 `HUIMAI_CODE_VERSION` 写入 `.next/BUILD_ID`；readiness 以 `X-Huimai-Code-Version` 返回该制品内值。预检要求它与完整 40/64 位 release SHA 精确一致，不能用运行时环境变量伪装成另一个版本。

值守检查：

```bash
curl -fsS http://127.0.0.1:3000/api/health/live
curl -fsS -D - http://127.0.0.1:3000/api/health/ready
```

readiness 返回 503 时，先查服务端日志，再依次核对迁移目录、SQLite、数据卷权限、剩余空间和 FFmpeg/FFprobe。不要通过修改健康接口让故障实例强行接流量。

### 运维告警台

工作人员登录后打开 `/admin/ops`。页面每 30 秒聚合一次以下门禁：

- 本机 readiness、磁盘和 FFmpeg；
- 持久任务待执行数、最老等待时间、过期租约和 24 小时终态失败；
- 模型 attempt 的 24 小时失败率、429、欠费、5xx 和 fallback；
- 成功 attempt 的可审计真实成本覆盖率与已知成本；成本为 `null` 表示尚未闭环，不计入覆盖率，也不能按 0 计费；
- 当前 release 绑定的备份验证证据、异机回执和对应正式 manifest 的新鲜度；不再把目录 mtime 或“最近有一个 manifest”当成有效备份。

出现“阻止放行”时先摘流或停止新增邀请，不得只清告警。可按容量评估配置 `HUIMAI_ALERT_PENDING_JOBS`、`HUIMAI_ALERT_PENDING_AGE_MS`、`HUIMAI_ALERT_MODEL_FAILURE_RATE`、`HUIMAI_ALERT_RATE_LIMIT_COUNT`、`HUIMAI_ALERT_BACKUP_AGE_MS` 和 `HUIMAI_ALERT_DAILY_COST_USD`；未配置时使用适合 5–10 家单实例内测的保守默认值。当前告警入口是站内值守台，部署平台仍应对 readiness、备份命令退出码和进程异常配置站外通知。

## 6. 在线备份

备份工具使用 SQLite 在线 backup API，不直接复制活动中的 `sqlite.db`。正式数据库快照只被复制到隔离临时目录后执行 `PRAGMA integrity_check` 与 `foreign_key_check`，验证过程不会打开正式备份数据库，避免 WAL/SHM 副文件污染。所有内容先写入 `*.part`；只有隔离数据库校验、媒体复制、精确文件集和逐文件 SHA-256 清单全部成功，才原子改名为正式备份目录。

2026-07-16 14:53 的第一轮本地演练曾因验证直接打开正式 WAL 数据库，在完成目录中留下 4 个未入清单的 WAL/SHM 文件，该证据已经作废。20:39 修复后本地备份为 504 个文件、851,494,814 字节，manifest SHA-256 为 `faaecb1f80d7102d1667fd4aa92bc9df289e254ed3c2a984fd7a4e7f8c948013`；它证明修复后的本地组件能力，但没有真实冻结窗口或异机回执，不能计入生产 RPO。详情见 `产品文档/16_备份恢复演练记录_2026-07-16.md`。

`uploads/` 和 `output/` 是逐文件复制，不具备数据库事务语义。脚本发现目录列表或文件元数据在复制期间变化时会失败关闭，但这不能替代全量写冻结。为保证 SQLite 记录与媒体文件属于同一业务时间点，每次完整备份前必须进入维护/只读窗口：从入口停止注册、资料/项目/脚本/指标/发布记录等数据库写入以及上传、删除、生成、TTS、合成等媒体写入；停止调度器和 worker 领取新任务；等待所有已开始写入排空；随后冻结全部 SQLite、`uploads/`、`output/` 写入，再开始备份。只读查询可以继续。必须记录冻结开始、任务排空、备份结束和解冻时间；若无法证明整个复制窗口没有任何一类写入，该备份不算有效的跨介质一致性或 RPO 证据。

### 手工执行

在已安装生产依赖、且能同时访问数据卷和独立备份卷的发布目录运行：

```bash
APP_DATA_DIR=/data BACKUP_DIR=/backups pnpm ops:backup
```

也可以显式指定路径：

```bash
node scripts/backup-data.mjs --source /data --destination /backups
```

容器内的等价命令（前提是镜像包含运维脚本且 `/backups` 已独立挂载）：

```bash
docker run --rm \
  --env-file /host/huimai-production.env \
  -v /host/huimai-data:/data:ro \
  -v /host/huimai-backups:/backups \
  verified-huimai-image@sha256:... \
  node /app/scripts/backup-data.mjs --source /data --destination /backups
```

这是一次性专用维护任务，不是长期 Web 容器；执行前 Web/worker 必须已经摘流、排空并冻结全部写入。不要用 `docker exec` 临时给长期 Web 放开备份卷写权限。

成功输出类似：

```text
备份完成：/backups/backup-2026-07-16T10-00-00-000Z-123
文件：123，字节：4567890
```

只有同时满足以下条件才算成功：

- 命令退出码为 0；
- 最终目录名不以 `.part` 结尾；
- 目录内存在 `sqlite.db` 和 `manifest.json`；
- `uploads/`、`output/` 源目录真实存在，且正式目录没有清单外文件、符号链接、特殊文件、`*-wal`、`*-shm` 或 `.part*`；
- 日志明确显示“备份完成”。

失败留下的 `*.part` 只用于排查，恢复工具会拒绝使用。确认没有仍在运行的备份进程后才能人工清理。“备份完成”只代表本地正式目录生成成功，尚未满足冻结窗口和异机副本放行证据。

### 绑定冻结窗口与异机版本回执

每份生产备份同步到异机/对象存储后，存储任务必须输出真实、不可变的版本回执，精确包含 `schemaVersion`、`provider`、`objectKey`、`versionId`、`manifestSha256`、`syncedAt`。占位值、可变对象无版本号、只记录上传成功日志或同机复制均不合格。

在专用证据目录执行：

```bash
pnpm ops:verify-backup -- \
  --backup /backups/backup-2026-07-16T10-00-00-000Z-123 \
  --evidence /release-evidence/backup-verification.json \
  --receipt /release-evidence/offsite-backup-receipt.json \
  --freeze-start 2026-07-16T09:55:00.000Z \
  --freeze-end 2026-07-16T10:10:00.000Z
```

验证器会重新检查正式目录精确文件集、逐文件大小/SHA、SQLite 完整性与外键；要求冻结窗口覆盖 manifest `createdAt` 且不超过 2 小时，异机 `syncedAt` 晚于备份，并生成只写一次的 evidence JSON 与 SHA-256。把 evidence 与 receipt 文件及各自 SHA 只读挂载到 Web，分别配置 `HUIMAI_BACKUP_EVIDENCE_FILE/SHA256`、`HUIMAI_BACKUP_OFFSITE_RECEIPT_FILE/SHA256`。预检会再次交叉验证；不得手写或伪造 `writesFrozen: true`。

### 调度与保留

- 每 6 小时执行一次完整备份；每次发布前再执行一次。调度流程必须自动编排“进入维护/只读窗口 → 停止取新任务并排空 → 记录冻结开始并冻结全部 SQLite/`uploads`/`output` 写入 → 一次性维护任务备份 → 异机同步并取得版本回执 → `ops:verify-backup` → 记录冻结结束 → 解冻”，不能只定时调用备份脚本。
- 邀请内测前 7 天保留所有 6 小时备份；之后至少保留 7 份日备份和 4 份周备份。
- 每一份 6 小时备份完成后立即同步到异机或对象存储，并把同步成功纳入同一轮调度状态；同一宿主机、同一磁盘上的目录不能算灾备。任何一份未同步都表示 6 小时整机灾难 RPO 尚未满足。
- 删除旧备份前先确认最新备份、上一份备份和一份异机备份都能通过恢复校验。脚本本身不会自动删除历史备份。
- 备份所需临时空间接近本次备份总量；执行前至少预留数据体积 1.2 倍的额外空间。

调度器必须保存退出码并告警：连续一次备份失败就通知值守人，连续两次失败暂停新增邀请。

## 7. 恢复到全新目录

恢复工具没有覆盖开关，且只接受不存在的目标目录。它会要求正式备份目录与 manifest 精确一致，拒绝清单外文件、符号链接、特殊文件和残留 WAL/SHM；校验每个文件的大小和 SHA-256，并只在隔离临时副本上检查源/目标 SQLite 完整性与外键。复制到新的 `*.part-*` 目录后再次校验，创建真实空 `uploads/`/`output/` 目录并最后原子改名。恢复过程不得直接打开正式备份数据库。

绝不直接向当前 `/data` 恢复，也不要先删除当前数据目录来“腾位置”。标准流程：

1. 摘流并停止所有 SQLite、`uploads/`、`output/` 写入，停止 worker 取新任务并排空已开始任务，记录故障时间；只暂停上传、生成和合成不足以覆盖账号、项目、指标、发布记录或删除等写入口。
2. 选择故障前最近一份没有 `.part` 后缀、已通过 `ops:verify-backup` 且异机版本回执与 manifest SHA 精确一致的备份；不能仅按目录 mtime 或“能看到 manifest”选择。
3. 恢复到全新挂载点或全新目录：

   ```bash
   node scripts/restore-data.mjs \
     --backup /backups/backup-2026-07-16T10-00-00-000Z-123 \
     --destination /data-restored-20260716
   ```

   从发布目录也可执行：

   ```bash
   pnpm ops:restore \
     --backup /backups/backup-2026-07-16T10-00-00-000Z-123 \
     --destination /data-restored-20260716
   ```

4. 用与正式部署相同的 release 镜像在独立端口启动，并挂载新的数据目录。若只做源码树诊断，可用下列命令，但它不能替代最终 Docker/standalone 制品演练：

   ```bash
   APP_DATA_DIR=/data-restored-20260716 PORT=3100 pnpm start
   ```

5. 先检查 `/api/health/live` 与 `/api/health/ready`，再完成以下人工验收：管理员登录、测试商户登录、项目列表、一个历史项目素材读取、一个历史成片下载、一次不计费的最小链路检查。
6. 记录所用备份、恢复耗时、校验结果和数据缺口。全部通过后，才把流量或数据卷挂载切到新目录。
7. 原数据目录保持只读并保留到本次事故复盘结束。任何回切也必须先摘流，严禁两个实例同时写同一 SQLite。

如果恢复失败，保留 `*.part-*` 目录和日志用于排查，换一个全新目标目录重试；不得绕过清单或 SQLite 完整性校验。

## 8. 发布与回滚 SOP

每次邀请内测发布严格按以下顺序执行：

1. 进入维护/只读窗口，停止全部 SQLite、`uploads/`、`output/` 新写入和 worker 取新任务，等待已开始的数据库及媒体写入全部排空，并记录冻结时间。
2. 由一次性维护任务执行发布前备份并同步异机，取得真实版本回执；运行 `ops:verify-backup` 绑定正式 manifest、冻结窗口和回执 SHA。Web 的备份卷始终保持只读。
3. 确认发布流水中的锁定依赖安装、lint、全量 TypeScript、测试、镜像构建与容器冒烟全部绿灯；任一步未执行或失败都不得手工跳过后推送镜像。
4. 记录当前镜像摘要、完整 Git 提交/BUILD_ID、迁移版本（当前 `0026`、共 27 条）、数据备份路径、manifest/evidence/receipt SHA 和冻结窗口。
5. 部署新版本；迁移失败即停止，不修改数据库来强行跳过迁移。
6. liveness 通过后检查 readiness、`no-store` 和 `X-Huimai-Code-Version`，再在运行容器中执行生产 `preflight-beta.mjs`；响应头与完整 release SHA 不一致、readiness 或预检未通过都不得接流量。
7. 使用专用测试商户完成登录、上传、脚本生成、媒体生成、合成和下载冒烟，并确认 202 轮询与同幂等键恢复正常。
8. 恢复邀请用户流量，至少观察 30 分钟错误率、任务失败、429/5xx、`submission_uncertain`、磁盘和单条成本。

需要回滚时，先判断旧代码是否与已经迁移的 schema 向后兼容。无法确认时，不允许旧代码直接连接已迁移数据库；应把发布前备份恢复到全新目录，用旧镜像 + 新恢复目录在独立端口验证，再切换流量。

## 9. 演练与留证

- 首次邀请用户前必须完成一次真实备份与恢复演练。
- 此后每月至少做一次随机备份恢复；任何数据库迁移发布前后各做一次。
- 演练不能止于 `integrity_check`：必须启动隔离实例并完成人工读取验收。
- 记录故障发现、恢复开始、恢复完成、人工验收、切流/回切的连续时间点，以及镜像摘要、Git 提交、备份目录、文件数/字节数、退出码、readiness、数据缺口和问题负责人；只有连续窗口才能作为完整 RPO/RTO 证据。
- 演练证据进入发布记录；未留证视为未完成。

## 10. 当前能力限制

- 备份调度、异机同步、保留清理和失败通知需要部署平台或系统定时任务承接；脚本负责生成可校验备份并验证冻结窗口/异机版本回执，但不能证明运维人员真的冻结了写入，真实证据不得伪造。
- SQLite 与媒体目录不是同一事务快照；没有在整个备份窗口冻结全部 SQLite、`uploads/`、`output/` 写入时，备份可能业务上不一致，不能计入 RPO/RTO 达标证据。
- 当前方案面向单实例 SQLite。需要多副本、跨地域容灾或公开付费规模前，应迁移到具备托管备份和高可用能力的数据库/对象存储方案。
- 当前任务 worker、HTTP 限流桶、上传字节预留锁和部分评测租约都以单进程/单机为运行边界。未完成共享队列、共享限流、跨进程锁与高可用存储前，严禁水平扩容。
- liveness/readiness 只负责本机基础依赖，不替代供应商 429、欠费、调用延迟、任务队列和成本告警。
