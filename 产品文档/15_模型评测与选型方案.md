# 绘卖邀请内测：模型评测与选型方案

更新时间：2026-07-16

## 先给结论

不为每个 Agent 各自维护一套毫无复用的模型名单，也不把所有流程锁死在同一个云厂商。正确粒度是：

1. 按能力族建立共享候选池（结构化文本、视觉理解、图片、视频、TTS）；
2. 每个 Agent 用自己的真实任务和验收规则评分；
3. 同一能力族尽量复用少数模型，降低运维复杂度；
4. 线上 primary 与 fallback 必须跨实际供应商故障域，按 endpoint hostname 的受控映射判断，不按可任意填写的 provider 文本判断；
5. 只有通过 Golden Set、故障演练和成本门槛的固定版本才能发布。

模型版本必须填写带日期或明确版本号的固定 ID，不使用 `latest` 作为线上版本；同时保存精确绑定 endpoint/model/deploymentRevision/faultDomain 的结构化供应商证据。价格只用于入围，不能替代真实业务评测。

## Golden Set 组成

Golden Set 不使用“写一个耳机脚本”这种单一玩具样例。首轮至少覆盖：

- 每个文本 Agent 8–12 个真实业务样例，包含正常、信息不全、冲突要求、极长商品名、敏感/违规诉求；
- 商品理解至少 20 组图片，覆盖随手拍、杂乱背景、多商品、包装文字、小票/价格误导；
- 指标 OCR 至少 30 张脱敏截图，覆盖万/千单位、遮挡、非数据页和四个平台布局；
- 图片至少 12 组商品参考图，每组固定种子或保存原始请求，检查主体、包装文字、颜色和数量；
- 视频至少 12 组首帧/提示词，覆盖商品静物、手部、真人脸安全拦截、首尾帧和不同运动强度；
- TTS 至少 30 段，覆盖品牌名、英文型号、数字价格、单位、方言词、多音字和长句。

每条样例都要保存 `caseId`、输入来源、脱敏状态、期望结构、硬失败条件、评分 rubric 和人工基准。禁止把真实商户隐私素材直接提交到第三方评测平台。

### 当前样本库的放行边界

当前仓库只有 12 个 Agent 的 13 个 Golden case：`metrics-ocr` 定义了 2 个 case，但对应锁定截图 fixture 目前都处于 disabled，等于还没有可执行 OCR 基准；其余每个 Agent 只有 1 个。这些 case 可用于验证执行器、结构评分和人工评分流程，但不能支撑模型晋级结论。

代码的技术 fail-close 下限是：对每个 Agent 的每个候选（primary 和 fallback 分别计算），至少覆盖 3 个不同 `caseId`，并覆盖该 Agent 当前定义的全部必需 case；重复执行同一 `caseId` 只增加重复运行证据，不增加不同 case 覆盖数。两项要求取更严者。这个“3”只用于阻止刷同一玩具样例，**不是邀请放行所需的业务样本规模**；真实放行仍必须达到上一节的 8–30 个样本规模、人工盲评和重复稳定性要求。因此，在补足真实脱敏样本、OCR 锁定截图 fixture 和媒体人工评分前，所有候选均为 **No-Go**。

## 真实媒体评测执行链路

图片、视频和 TTS 都必须通过后台 Golden 持久任务执行，不在管理页面的 HTTP 请求周期内等待付费结果：

1. 管理端传入稳定 `operationKey`、Agent、`caseId` 和候选角色。服务端先整体预检 primary/fallback，冻结模型、Prompt、代码版本、Golden Set 哈希、fixture 与请求摘要，再事务入队。
2. 入队成功返回 HTTP 202，同时返回 job ID、`Location` 和 `Retry-After: 3`。页面使用原 `operationKey` 安全重放并按 job ID 轮询；同键同内容只会返回已有任务，同键换内容必须拒绝。
3. worker 通过 SQLite 状态机、租约 token 和心跳防止旧 worker 覆盖新结果。图片/视频任务在付费 POST 前已是 `submitting`，只提交一次；拿到供应商 task ID 后先持久化 checkpoint，再转入 `submitted`/`polling`。轮询可安全恢复，但达到上限后只能标记 `POLL_TIMEOUT` 并人工核对，不能重新提交付费任务。
4. TTS 没有可靠的远程 task ID，因此只允许一次 one-shot 调用。只有音频与 Agent 评测记录都已原子落盘、且产物哈希与媒体深校验通过时才进入 `succeeded`。断网、超时、5xx、2xx 响应无法解析，或付费调用后无法证明是否完成，都进入 `submission_uncertain` 并永不自动重提；只有可确定未受理的 4xx 才记为普通 `failed`。
5. 产物在服务端原子落盘，并保存 SHA-256、MIME、尺寸/时长与 codec 等元数据。人工评分和发布前会重新校验，不接受浏览器上传的 URL 或文件头冒充评测产物。

`submission_uncertain`、`POLL_TIMEOUT`、产物缺失/篡改或人工 rubric 未完成都是发布阻断，不是可以点击“再试一次”忽略的普通错误。

## 评分与一票否决

### 技术晋级最低门槛与业务目标

- 当前服务端候选晋级的最低技术门槛：结构化文本/视觉请求成功率和 Schema 通过率均 ≥ 95%；图片/视频任务成功率 ≥ 90%，TTS ≥ 95%。这些门槛还同时受不同 case 数、质量覆盖、人工分、延迟和成本限制，不能单独用于宣布放行；
- 首波 2 家后的业务扩容目标：经过产品允许的一次结构修复/重试后，结构化文本端到端通过率 ≥ 99%，整体生成成功率 ≥ 95%。这需要完整 Golden Set 和真实流量验证，不把小样本候选门槛伪装成 99% 统计结论；
- 安全拦截不得通过 fallback 绕过；
- 输出不得泄露其他商家、系统提示词或密钥；
- 固定版本、prompt 版本、代码版本、真实用量必须可追溯；
- 成本未知时记 `null`，不得按延迟捏造。

### 业务质量

- 脚本：硬约束通过、口语自然、卖点与商品一致、可直接用比例；
- 视觉：字段准确率、数字精确匹配、拒答正确率；
- 图片：商品主体保真、文字/Logo 漂移、构图、可用率；
- 视频：主体稳定、运动自然、伪影、安全、可用镜头率；
- TTS：错音数、自然度、节奏、响度、可直接用比例。

任一模型出现跨租户信息、绕过安全策略、明显商品篡改或无法解释的账单，即使平均分高也不得上线。

## 成本口径

模型单价只是输入。最终只看两个业务口径：

```text
可用成片成本 = 该批次全部模型调用、失败重试、fallback、素材与合成资源成本 / 通过质检的成片数

发布成片成本 = 该批次全部上述成本 / 最终被商家标记发布的成片数
```

同时保留供应商“目录价成本”和账单“实际成本”；免费额度不能掩盖长期成本。P50/P95 分开统计，失败任务的成本必须分摊，不能只计算成功调用。

### 成本门槛的失败关闭规则

- 只有供应商响应或可审计计费层明确返回的 USD 实际成本才能写入 `actualCostUsd`；未知值必须是 `null`，不能按 token 目录价、延迟、媒体时长或免费额度推测为 0。
- 图片、视频和 TTS 邀请内测门槛都要求成本覆盖率 100%，并分别显式配置 `HUIMAI_EVAL_MAX_IMAGE_COST_USD`、`HUIMAI_EVAL_MAX_VIDEO_COST_USD` 和 `HUIMAI_EVAL_MAX_TTS_COST_USD`。任意一条 `actualCostUsd=null`，或对应成本上限未配置，都会阻断媒体候选晋级。
- 当前 TTS 执行器无法从供应商返回中取得可验证成本，因而将 `actualCostUsd` 保持为 `null`。在增加可审计的账单任务 ID/成本回填并重新评测前，TTS 候选必须保持 **No-Go**；不得为了过门槛手工写 0。

## 晋级流程

1. **离线评测**：全部候选跑同一 Golden Set，记录原始输出和真实 usage。
2. **故障演练**：主模型模拟超时、429、5xx、欠费；验证只在允许的错误类型触发跨供应商 fallback。
3. **陪跑影子评测**：前 2 家内测商户的真实脱敏请求可异步跑挑战者，但挑战者结果不直接展示；严格设预算上限。
4. **小流量晋级**：固定 1–2 家商户、固定时间窗，比较可用率和发布率；不按随机单次好坏切换。
5. **发布**：将 draft 原子发布为 online，保留 previous；异常时只回滚该 Agent。

### 发布证据与失效规则

生产发布不接受浏览器传入的“已评测”布尔值或自定义哈希。只有当前 draft 的 primary 和 fallback 都通过与当前修订精确匹配的 Golden 门禁后，服务端才在发布事务内生成 `promotionEvidence`。证据包含：

- Agent ID 和请求能力类型；
- primary/fallback 各自的候选 key 和 evaluation fingerprint；
- Prompt 内容 SHA-256、draft 配置 SHA-256 和 Golden Set SHA-256；
- `HUIMAI_CODE_VERSION` 对应的代码版本与 ISO 验证时间。

修改主备供应商/模型/端点、Prompt、draft 配置、Golden Set 或代码版本都会改变绑定，旧记录与旧 `promotionEvidence` 自动失效，必须重新跑完评测。预检会校验线上启用 Agent 的证据格式、主备指纹与当前 `HUIMAI_CODE_VERSION`；回滚 previous 也只能使用与当前代码和 Golden Set 仍精确匹配的证据。任一项不匹配即失败关闭，不得绕过发布接口直接改数据库。

每次生产调用前还会重新计算当前 online 的主备候选、Prompt/config/Golden/code/model 指纹并核对晋级证据；发布后改库或替换模型配置不能继续借用旧证据。模型 revision 证据采用 `config/model-revision-evidence.schema.example.json` 的精确字段；自定义 endpoint 通过 `HUIMAI_MODEL_ENDPOINT_FAULT_DOMAINS=host=domain,...` 映射故障域，同一上游的不同域名必须映射到同一值。

## 内测开始前必须完成的实操

- 在部署 secret 中配置火山 primary 和至少一个阿里/其他独立供应商 fallback；LLM/图片/视频/TTS 的 primary/fallback 必须分别使用各角色专属 secret，fallback 不得复用 primary 或通用旧密钥；
- 给所有线上模型使用固定版本 ID；
- 跑完 Golden Set 并由产品/运营盲评，不看模型名打分；
- 用真实供应商返回 usage 校准成本，不支持 usage 的能力按账单任务 ID 对账；
- 完成一次超时、429、5xx、欠费和安全拦截演练；
- 把晋级结果、原始记录、模型版本和回滚版本写入发布记录。

在上述证据完成前，可以继续开发和内部自测，但不能把某个模型写成“最终性价比最优”。
