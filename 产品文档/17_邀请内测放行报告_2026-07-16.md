## 3. 模型选型决策

需要按 Agent 真实任务做评测，但不应为每个 Agent 维护一套完全独立的模型。放行粒度应当是“**按能力族共享候选池，按 Agent 分别验收**”：

- 结构化文本、视觉/OCR、图片、视频和 TTS 分别建立候选池；
- 同一能力族尽量收敛到少量供应商侧可核验的不可变 revision/deployment，降低运维复杂度；
- 每个 Agent 仍使用自己的真实样例、硬失败条件和人工 rubric 独立晋级；
- online primary 与 fallback 必须跨供应商故障域；仅幂等任务中显式分类的 transient/parse 错误，或已知可接受且能证明请求未被受理的 billing/quota 错误可以降级，safety、config、其余 client 4xx、unknown 和 `submission_uncertain` 必须 fail-close；
- 只有主备都通过当前代码、Prompt、Golden Set 和成本门槛绑定的证据，才能发布。

当前只能确认评测机制已具备，**不能宣布火山、阿里或其他任何候选已是最终性价比方案**。具体候选池、Golden Set 和晋级规则见 `产品文档/15_模型评测与选型方案.md`。

## 4. 放行前必须严格按顺序执行

以下步骤不得并行跳级或用人工改库绕过；上一步未留证，不得进入下一步。

### 第 1 步：固化 release 边界

1. 将当前上线范围收口到明确的 release 分支或提交，确认不包含本地数据、媒体、`.env*`、Git 历史、版本归档和清单列明的敏感研发归档；Next.js 运行时必须追踪的源码不作“零源码”承诺。
2. 在干净 clone 中完成锁定依赖安装、全量测试、TypeScript、lint、Next build 和 standalone 敏感内容/体积校验。
3. 在同一提交上跑通 GitHub CI 与 Docker build/smoke，记录 Git commit、迁移版本、镜像摘要和全部作业链接。

**通过标准：** release 提交可从干净环境复现，远程 CI 和容器冒烟全绿，已留存不可混淆的镜像摘要。

**当前状态：已完成。** 证据绑定提交 `fa266c5ab6d986e2f67d8180583094892dd3e78a`、CI `29523621458`、Docker `29523668544`、上述不可变镜像摘要与发布证据 artifact；进入第 2 步不代表允许跳过第 2–6 步。

### 第 2 步：准备真实生产环境

1. 确定 HTTPS 域名、持久数据卷、独立备份卷和只启动单个 Web/worker 实例的部署约束；Web 长期只读挂载备份卷，只有一次性专用维护任务在冻结窗口短时以读写方式挂载。
2. 注入不同的后台强密码与 session secret，配置真实邀请邮箱名单，保持公开注册关闭。
3. 确定运营主体、隐私/客服联系方式、数据保留期限、实际模型服务商披露和 AIGC 服务提供者标识；在部署平台落实与披露一致的日志保留、删除请求和处理留证流程。
4. 落实备份每 6 小时调度；每轮先进入维护/只读窗口，停止接收新写入和 worker 取新任务，排空现有任务，冻结全部 SQLite、`uploads/`、`output/` 写入并留存冻结/解冻时间，再由专用维护任务执行备份、异机/对象存储同步、保留清理与失败站外通知；取得真实对象版本回执后运行 `pnpm ops:verify-backup`，绑定正式 manifest、冻结窗口、异机版本和各文件 SHA-256。
5. 同时安装仓库 Nginx vhost 与 proxy snippet（或平台等价策略），默认 body 2 MiB，仅在上传路由放宽至 21/81/161 MiB；上传入口执行连接/速率限制，应用端口只对反代开放，反代覆盖 `X-Real-IP` 并清除来访 CF/XFF 等来源头。对部署机 `nginx -T` 完整输出计算 SHA-256；运行 `pnpm ops:verify-edge`，证明材料路由 `max-1` 正向对照非 413，`max+1` 的 `Content-Length` 与 chunked 均为 413。

**通过标准：** 所有环境、法务和运维配置都是真实值，无示例文案、共享弱凭据或本地临时路径；原始请求体入口门禁及冻结全部写入的备份编排均已实测留证。

### 第 3 步：配齐模型主备候选

1. 按能力族为所有将启用的 Agent 配置供应商侧不可变或已锁定且可核验 revision/deployment 的 primary 和 fallback；每个端点只绑定一个实际模型，不允许不同 `model/visionModel` 或线上按模式改写模型 ID；`latest`、展示别名或可随时被重新指向的 deployment 名称本身都不算固定版本证据。结构化 revision 证据必须精确绑定 endpointHost、model、deploymentRevision、faultDomain、immutable、evidenceType、sourceReference 和 issuedAt。
2. 主备使用不同实际供应商故障域；以 endpoint hostname 的内置/部署映射判断，不以可任意填写的 provider 名称判断。同一上游的不同域名必须映射到同一故障域；端点只允许 HTTPS 且在生产主机白名单内。
3. 真实密钥只经部署 secret 注入，控制面只保存 `secretRef`；primary/fallback 必须分别使用各角色专属 secret，不允许 fallback 回退复用 primary 或通用旧密钥。
4. 先用最小受控请求校验连通、供应商实际 revision/deployment 和账单归属，保存控制台/API 回执或等价证据，不因“可调通”或配置字符串看似带版本号直接发布。

**通过标准：** 所有启用 Agent 的主备候选配置完整、上游不可变版本可核验、故障域独立，且无明文密钥。

### 第 4 步：补齐 Golden Set 并执行真实付费评测

1. 按 `产品文档/15_模型评测与选型方案.md` 补齐真实、脱敏、可重现的文本、视觉/OCR、图片、视频与 TTS 样本和 rubric。
2. 达到方案规定的完整规模：文本每 Agent 8–12、商品理解 20、OCR 30、图片 12、视频 12、TTS 30。代码中的每候选至少 3 个不同 `caseId` 只是禁止重复刷题的 fail-close 下限，不能替代这些业务样本。
3. 主备候选跑同一批样本，保存原始输入/输出、供应商返回且可核验的 revision/deployment、usage、延迟、失败原因和产物哈希。
4. 由产品/运营在不看模型名的前提下完成人工盲评；完成前不得发布媒体候选。
5. 用供应商返回或可审计账单任务回填实际成本；无法验证的成本保持 `null`。先补齐 TTS 的可审计成本链路，再评测与晋级。
6. 按“通过质检成片”和“商户最终发布成片”分别统计 P50/P95 成本，失败、重试和 fallback 成本不得剔除。

**通过标准：** 质量、安全、覆盖、成本和人工 rubric 全部达标，每个将上线候选都产生与当前修订精确绑定的服务端晋级证据。

### 第 5 步：完成供应商故障演练

1. 依次演练 primary 超时、429、5xx、欠费和安全拦截。
2. 验证只有幂等任务中显式分类的 transient/parse 错误，或已知可接受且能证明请求未被受理的 billing/quota 错误会进入跨供应商 fallback；safety、config、其余 client 4xx、unknown 和 `submission_uncertain` 必须 fail-close，本地模板或硬编码模型也不得绕过该边界。
3. 对图片、视频与 TTS 单次付费提交，验证超时或响应损坏不会自动重复扣费，`submission_uncertain` 会阻断晋级并进入人工对账。
4. 保存 requestId、attempt、fallback 原因、候选版本、账单任务、最终状态和运维页告警证据。

**通过标准：** 所有故障类型的实际路由、不重复付费和失败关闭行为与设计一致，且可由日志、评测记录与账单交叉追溯。

### 第 6 步：执行真实生产 preflight 和发布冒烟

1. 用即将发布的镜像、真实数据卷、只读备份卷与生产环境变量在隔离端口启动，等待迁移、liveness 和 readiness 完成；readiness 的 `X-Huimai-Code-Version` 必须来自制品 `.next/BUILD_ID`，并与构建时注入的完整 40/64 位 release SHA 精确一致。

出现以下任一情况时，立即停止新邀请、暂停高成本任务，必要时摘流：

- 重复付费、账单无法对账、实际成本持续为 `null` 或超出已批准预算上限；
- `submission_uncertain` 或持久任务异常积压无法在值守时间窗内安全收口；
- 无法用当前镜像、数据备份和模型 previous 完成可验证回滚。

## 7. 已知边界

1. **只支持单机单实例。** SQLite、任务 worker、HTTP 限流桶、上传字节预留和部分租约不支持多进程协调。引入多副本前必须迁移至共享队列、共享限流、跨进程锁与高可用存储。
2. **运维告警主体仍是站内页。** readiness、备份命令失败、进程异常与账单异常还需部署平台配置站外通知。
3. **备份工具不等于灾备运营。** 14:53 的旧证据已经因 WAL/SHM 污染作废；20:39 记录也只证明修复后的本地组件。定时调度、异机同步、真实版本回执、`ops:verify-backup`、保留删除与失败通知必须由部署平台或系统任务承接；Web 长期只读挂载备份卷，每次备份前必须进入维护/只读窗口，排空任务并冻结全部 SQLite、`uploads/`、`output/` 写入。
4. **远程制品验证不等于公网网站上线。** CI、Linux 容器冒烟、GHCR 不可变摘要和结构化证据已经完成；只有真实服务器、域名/DNS/TLS、生产配置、持久卷、Nginx 边缘证据、生产 preflight 与外网业务冒烟同时通过，才可对外发送网址。
5. **模型治理机制不等于模型已选定。** 当前未完成真实付费评测、成本对账与故障演练，不得以供应商生态便利性替代业务评测结论。
6. **本报告只适用邀请内测。** 它不授权公开注册、公开付费、无人值守、水平扩容或公开投放。
